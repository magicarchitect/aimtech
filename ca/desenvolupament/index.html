01 — Què construïm

Quatre formats / de programari a mida.

No venem plantilles ni paquets tancats. Construïm el que el problema demana, amb l'stack que millor el resol. Si el problema cap en un full de càlcul ben fet, t'ho diem i ens estalviem un projecte.

// format_01 Web

Aplicacions web.

SaaS, dashboards interns, portals per a clients, intranets. SSR quan importa el SEO o el time-to-first-paint, SPA quan el flux és d'aplicació pura. Disseny cuidat, performance mesurable.

React Next.js SSR / SPA Vanilla JS
// format_02 Mòbil

Apps mòbils.

iOS i Android des del mateix codi base amb React Native o Flutter quan el cas ho permet. Natiu (Swift/Kotlin) si el rendiment o la integració amb el sistema operatiu ho justifiquen.

React Native Flutter Swift / Kotlin PWA
// format_03 CMS

Gestors de continguts.

Quan WordPress es queda curt o sobra per al teu cas. Un CMS fet a mida amb només els camps, rols i fluxos que el teu equip necessita — sense plugins aliens que es trenquen quan s'actualitzen i sense formularis que ningú entén.

Headless Rols & permisos Editor a mida Workflows
// format_04 Backend

Backend i APIs.

REST o GraphQL, integracions amb sistemes que ja tens (CRM, ERP, comptabilitat), passarel·les amb serveis externs, jobs en cua, webhooks. Amb tests, logs estructurats i un panell de monitoring real, no un grep en producció.

Node.js / Python REST / GraphQL PostgreSQL Integracions
02 — Mètode

De la idea a producció / en passos curts.

Res d'specs de 80 pàgines que quan se signen ja són obsoletes. Treballem per iteracions curtes amb lliuraments que es poden tocar — i decisions que es prenen amb alguna cosa funcionant al davant, no amb un PowerPoint.

01 Discovery

Entendre el problema real.

Una o dues sessions amb qui pateix el problema. En sortim amb l'abast del MVP, les decisions tècniques grans (quin stack, què s'integra amb què) i una proposta tancada de la primera iteració. Si després del discovery no hi ha cas, t'ho diem.

02 MVP

Quelcom que es pugui fer servir.

Sprint de 2–4 setmanes que acaba amb un sistema funcional desplegat en un entorn real, no en un Figma. L'abast s'acota al que demostra valor; el bonic i el perifèric van en sprints posteriors.

03 Iteració

Millorar amb ús real.

El feedback dels primers usuaris reordena el backlog. Cada iteració afegeix el que fa mal, no el que sonava bonic al principi. Revisió setmanal amb demo, mètriques d'ús i decisió sobre què entra al següent sprint.

04 Producció

Lliurar de debò.

CI/CD, backups, logs estructurats, observabilitat, pla d'incidents. El codi queda al teu repositori amb documentació de desplegament perquè el teu equip (o el següent proveïdor) el pugui mantenir sense trucar-nos a les 3 del matí.

03 — Stack adaptat

L'stack s'escull per la necessitat, / no per moda.

No hi ha una resposta única bona. Hi ha decisions raonables segons el cas. Aquestes són les eines amb què treballem cada dia; si la teva organització ja té una preferència raonable, la respectem.

// stack_01 Frontend

Frontend i UX.

React + Next.js per a aplicacions grans amb SEO. JS vanilla quan el lloc és estàtic i no compensa el framework. CSS sense frameworks pesats: control real sobre el disseny i rendiment que no necessita explicació.

React Next.js TypeScript CSS vanilla
// stack_02 Backend & dades

Backend i dades.

Node.js o Python segons l'ecosistema del problema. PostgreSQL com a base per defecte — obert, robust i sense sorpreses en facturació. Redis per a caché i cues. Tests, migracions versionades i observabilitat des del dia u.

Node.js Python PostgreSQL Redis FastAPI
// stack_03 Mòbil & cloud

Mòbil i cloud.

React Native o Flutter quan un sol codi per a iOS i Android és raonable. Desplegament a AWS, GCP, Azure o en infraestructura pròpia: el codi no es casa amb un proveïdor que demà canvia els preus. Contenidors, CI/CD i backups verificats.

React Native Flutter Docker AWS / GCP CI/CD
04 — Quan té sentit

Quan sí / i quan no.

Un desenvolupament a mida no sempre és la resposta. De vegades el correcte és comprar una eina feta i configurar-la bé. T'ajudem a veure-ho clar abans de signar res.

Té sentit quan:

  • Les eines estàndard es queden curtes — el flux és prou específic perquè cap SaaS encaixi sense doblegar-lo fins a trencar-lo.
  • El procés és nuclear per al negoci — i dependre d'un proveïdor que demà puja preus o tanca el servei és un risc real.
  • El cost mensual de llicències supera amb marge el cost del desenvolupament en un horitzó raonable (24–36 mesos).
  • Necessites integrar diversos sistemes interns que cap SaaS del mercat coneix.
  • L'equip farà servir l'eina cada dia — invertir en una UX feta a la seva mida té retorn.

No compensa si:

  • Existeix un SaaS raonable que resol el 90% per menys del que costaria desenvolupar-lo i mantenir-lo durant dos anys.
  • L'ús serà esporàdic o el faran servir 3 persones un cop al mes — un full ben fet aguanta més del que sembla.
  • L'organització no té qui mantingui el sistema un cop lliurat, i tampoc hi ha pressupost per a suport continuat.
  • El problema encara no és clar — primer convé mapejar-lo bé amb qui el pateix abans de posar algú a programar.
