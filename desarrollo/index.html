01 — Qué construimos

Cuatro formatos / de software a medida.

No vendemos plantillas ni paquetes cerrados. Construimos lo que el problema pide, con el stack que mejor lo resuelve. Si el problema cabe en una hoja de cálculo bien hecha, te lo decimos y nos ahorramos un proyecto.

// formato_01 Web

Aplicaciones web.

SaaS, dashboards internos, portales para clientes, intranets. SSR cuando importa el SEO o el time-to-first-paint, SPA cuando el flujo es de aplicación pura. Diseño cuidado, performance medible.

React Next.js SSR / SPA Vanilla JS
// formato_02 Móvil

Apps móviles.

iOS y Android desde el mismo código base con React Native o Flutter cuando el caso lo permite. Nativo (Swift/Kotlin) si el rendimiento o la integración con el sistema operativo lo justifican.

React Native Flutter Swift / Kotlin PWA
// formato_03 CMS

Gestores de contenido.

Cuando WordPress se queda corto o sobra para tu caso. Un CMS hecho a medida con sólo los campos, roles y flujos que tu equipo necesita — sin plugins ajenos que se rompen al actualizar y sin formularios que nadie entiende.

Headless Roles & permisos Editor a medida Workflows
// formato_04 Backend

Backend y APIs.

REST o GraphQL, integraciones con sistemas que ya tienes (CRM, ERP, contabilidad), pasarelas con servicios externos, jobs en cola, webhooks. Con tests, logs estructurados y un panel de monitoring real, no un grep en producción.

Node.js / Python REST / GraphQL PostgreSQL Integraciones
02 — Método

De la idea a producción / en pasos cortos.

Nada de specs de 80 páginas que cuando se firman ya están obsoletas. Trabajamos por iteraciones cortas con entregas que se pueden tocar — y decisiones que se toman con algo funcionando delante, no con un PowerPoint.

01 Discovery

Entender el problema real.

Una o dos sesiones con quien sufre el problema. Salimos con el alcance del MVP, las decisiones técnicas grandes (qué stack, qué se integra con qué) y una propuesta cerrada de la primera iteración. Si después del discovery no hay caso, lo decimos.

02 MVP

Algo que se pueda usar.

Sprint de 2–4 semanas que termina con un sistema funcional desplegado en un entorno real, no en un Figma. El alcance se acota a lo que demuestra valor; lo bonito y lo periférico van en sprints posteriores.

03 Iteración

Mejorar con uso real.

El feedback de los primeros usuarios reordena el backlog. Cada iteración añade lo que duele, no lo que sonaba bonito al principio. Revisión semanal con demo, métricas de uso y decisión sobre qué entra en el siguiente sprint.

04 Producción

Entregar de verdad.

CI/CD, backups, logs estructurados, observabilidad, plan de incidentes. El código queda en tu repositorio con documentación de despliegue para que tu equipo (o el siguiente proveedor) pueda mantenerlo sin llamarnos a las 3am.

03 — Stack adaptado

El stack se elige por la necesidad, / no por moda.

No hay una respuesta única buena. Hay decisiones razonables según el caso. Estas son las herramientas con las que trabajamos a diario; si tu organización ya tiene una preferencia razonable, la respetamos.

// stack_01 Frontend

Frontend y UX.

React + Next.js para aplicaciones grandes con SEO. JS vanilla cuando el sitio es estático y no merece la pena el framework. CSS sin frameworks pesados: control real sobre el diseño y rendimiento que no necesita explicación.

React Next.js TypeScript CSS vanilla

Backend y datos.

Node.js o Python según el ecosistema del problema. PostgreSQL como base por defecto — abierto, robusto y sin sorpresas en facturación. Redis para caché y colas. Tests, migraciones versionadas y observabilidad desde el día uno.

Node.js Python PostgreSQL Redis FastAPI

Móvil y cloud.

React Native o Flutter cuando un solo código para iOS y Android es razonable. Despliegue en AWS, GCP, Azure o en infraestructura propia: el código no se casa con un proveedor que mañana cambia los precios. Contenedores, CI/CD y backups verificados.

React Native Flutter Docker AWS / GCP CI/CD
04 — Cuándo tiene sentido

Cuándo sí / y cuándo no.

Un desarrollo a medida no siempre es la respuesta. A veces lo correcto es comprar una herramienta hecha y configurarla bien. Te ayudamos a verlo claro antes de firmar nada.

Tiene sentido cuando:

  • Las herramientas estándar se quedan cortas — el flujo es lo suficientemente específico como para que ningún SaaS encaje sin doblarlo hasta romperlo.
  • El proceso es nuclear para el negocio — y depender de un proveedor que mañana sube precios o cierra el servicio es un riesgo real.
  • El coste mensual de licencias supera con margen el coste del desarrollo en un horizonte razonable (24–36 meses).
  • Necesitas integrar varios sistemas internos que ningún SaaS de mercado conoce.
  • El equipo va a usar la herramienta a diario — invertir en una UX hecha a su medida tiene retorno.

No compensa si:

  • Existe un SaaS razonable que resuelve el 90% por menos de lo que costaría desarrollarlo y mantenerlo durante dos años.
  • El uso será esporádico o lo van a tocar 3 personas una vez al mes — una hoja bien hecha aguanta más de lo que parece.
  • La organización no tiene quién mantenga el sistema una vez entregado, y tampoco hay presupuesto para soporte continuado.
  • El problema todavía no está claro — primero conviene mapearlo bien con quien lo sufre antes de poner a alguien a programar.
05 — Preguntas frecuentes

Desarrollo de software a medida / sin humo.

Las preguntas que suelen aparecer antes de invertir en una aplicación propia, un CMS a medida o una API interna.

¿Qué tipo de software a medida desarrolla Aimtech?

Aplicaciones web, apps móviles, gestores de contenido personalizados, integraciones entre sistemas y backends/APIs para empresas, entidades y administraciones públicas.

¿Cuándo conviene frente a un SaaS?

Cuando el proceso es nuclear, las herramientas estándar se quedan cortas, hay que integrar sistemas internos o las licencias empiezan a pesar más que una solución propia.

¿Con qué stack trabajáis?

Elegimos la tecnología según el problema: React, Next.js, JS vanilla, Node.js, Python, PostgreSQL, React Native, Flutter y cloud cuando aporta valor real.

¿Entregáis algo mantenible?

Sí: repo versionado, despliegue, documentación práctica, logs, backups y una arquitectura que tu equipo pueda entender cuando Aimtech no esté delante.
